# blxck.hub

Тестовая видеоплатформа с **синхронными комнатами совместного просмотра**.
Расчётно-графическая работа по дисциплине «Web-технологии» (СибГУТИ).

Пользователи загружают видео, смотрят их в общей ленте, а главное — создают
**комнаты**, где несколько зрителей смотрят ролик синхронно: ведущий управляет
плеером, плееры зрителей подстраиваются под него, рядом идёт живой чат с лайками
на сообщениях и вкладка «Вопрос/ответ».

## Стек технологий

| Слой | Технологии |
|------|-----------|
| Бэкенд | Python, Django 5, Django REST Framework, Django Channels (WebSocket) |
| Реал-тайм | Daphne (ASGI), Redis (channel layer) |
| База данных | PostgreSQL (на сервере), SQLite (локальная разработка) |
| Авторизация | JWT (djangorestframework-simplejwt), вход по одноразовому коду |
| Фронтенд | React, React Router, Vite, axios |
| Дизайн | Дизайн-система из макета Figma, тёмная тема, адаптивная вёрстка |
| Инфраструктура | Docker, docker-compose, Caddy (reverse proxy + TLS Let's Encrypt) |

## Возможности

- Регистрация и вход **без пароля** — по одноразовому коду на e-mail, JWT-сессии.
- Загрузка видео, лента, страница просмотра, потоковая отдача с поддержкой
  HTTP Range (перемотка).
- Профиль-канал: свои видео, управление, редактируемое «имя в чате».
- **Комнаты совместного просмотра**: синхронизация плеера по серверным часам,
  коррекция дрейфа, роль ведущего, ссылка-приглашение, гостевой доступ.
- Живой чат с лайками на сообщениях и вкладка «Вопрос/ответ» (вопросы, ответы,
  голоса) — поверх WebSocket.

## Быстрый старт — локальная разработка

Зависимости: Python 3.10+, Node.js 20+.

**Бэкенд** (SQLite + in-memory channel layer, никаких внешних сервисов):

```bash
cd app/backend
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
.venv/bin/python manage.py migrate
.venv/bin/daphne -b 127.0.0.1 -p 8000 config.asgi:application
```

**Фронтенд** (в отдельном терминале):

```bash
cd app/frontend
npm install
npm run dev
```

Откройте `http://localhost:5173`. Vite проксирует `/api` и `/ws` на бэкенд.
Одноразовые коды входа печатаются в лог бэкенда (console-почта).

## Запуск через Docker (весь стек одной командой)

```bash
docker compose up --build
```

Поднимет PostgreSQL, Redis, бэкенд и фронтенд; платформа доступна на
`http://localhost:8080`.

## Деплой на сервер

`deploy.sh` разворачивает платформу на сервере с уже настроенным Caddy-стеком
(`/opt/stack`). Запускать **на сервере**:

```bash
./deploy.sh --install      # развернуть
./deploy.sh --uninstall    # снести
```

Скрипт идемпотентен: синхронизирует исходники в `/opt/stack/blxckhub`, добавляет
сервисы в общий `docker-compose.yml`, дописывает поддомен в `Caddyfile`,
поднимает стек. TLS выпускается автоматически через Let's Encrypt.

## Переменные окружения

Полный список — в [.env.example](.env.example). Ключевые:

| Переменная | Назначение |
|-----------|-----------|
| `DJANGO_SECRET_KEY` | Секретный ключ Django |
| `DJANGO_DEBUG` | Режим отладки (`False` на сервере) |
| `DJANGO_ALLOWED_HOSTS` | Разрешённые хосты |
| `DATABASE_URL` | Подключение к БД; без неё — SQLite |
| `REDIS_URL` | Redis для channel layer; без него — in-memory |
| `EMAIL_BACKEND` | Доставка кодов: console (по умолчанию) или SMTP |
| `CORS_ALLOWED_ORIGINS` | Origin фронтенда |

Реальный `.env` в репозиторий не коммитится.

## Авторизация без пароля

Платформа не хранит пароли. Регистрация принимает e-mail, имя и фамилию; вход —
e-mail. В обоих случаях на почту отправляется 6-значный код (хранится в БД
хешированным, живёт 10 минут, одноразовый, блокируется после 5 неверных
попыток). После проверки кода выдаётся пара JWT-токенов: access (30 минут) и
refresh (7 дней).

## Тестирование

```bash
# Бэкенд — pytest (модели, REST, WebSocket-консьюмеры)
cd app/backend && .venv/bin/python -m pytest

# Фронтенд — Vitest (компоненты, утилиты, логика синхронизации)
cd app/frontend && npm run test
```

## Известные ограничения

- Передача роли ведущего другому участнику не реализована; при отключении
  ведущего комната остаётся на последнем состоянии.
- Refresh-токен хранится в `localStorage` — компромисс простоты против
  httpOnly-cookie (потребовала бы CSRF-защиты).
- Видео отдаётся как загружено, без транскодирования в несколько качеств.
